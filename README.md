# Финские Новости — бэкенд агрегатора

Реальный сервис, который тянет RSS из YLE, Helsingin Sanomat, Iltalehti и
Ilta-Sanomat и **пересказывает** заголовки и краткое содержание на русском
языке — не дословный перевод, а самостоятельная переформулировка сути через
Claude API, с обязательной ссылкой на оригинал у каждой новости.

## Быстрый старт

```bash
npm install
cp .env.example .env
# впишите в .env свой ANTHROPIC_API_KEY (см. ниже)
npm start
```

Сервер поднимется на `http://localhost:3000`. Первое обновление лент
запустится сразу при старте, дальше — каждые `REFRESH_INTERVAL_MINUTES` минут
(по умолчанию 15).

Проверить, что всё работает, без ожидания cron:
```bash
npm run refresh-once
```

## Как работает пересказ (src/aiRetell.js)

На вход модели идут только заголовок и краткое описание, которые сам сайт
уже публикует в открытом RSS-анонсе — не полный текст платной статьи.
Промпт явно требует от модели:
- не переводить дословно, а формулировать по-русски самостоятельно;
- не добавлять фактов, которых нет в оригинале;
- не копировать структуру и обороты речи исходного текста;
- уложиться в 2–3 предложения.

Статьи и результаты пересказа хранятся в SQLite (`DATABASE_PATH`, по
умолчанию `data/finskienovosti.db`). Уже сохранённые URL не пересказываются
повторно.

Перед вызовом AI новые RSS-анонсы дополнительно сравниваются с материалами
других источников за тот же день. Совпадение пропускается только при высокой
уверенности по заголовку и краткому содержанию. Решение сохраняется в журнале
`article_duplicate_log` и видно редактору в `/admin`; уже опубликованный архив
никогда не удаляется этой проверкой.

## Ключ Claude API

1. Зарегистрируйтесь на https://console.anthropic.com
2. Создайте ключ: https://console.anthropic.com/settings/keys
3. Впишите в `.env` → `ANTHROPIC_API_KEY`

По умолчанию используется `claude-haiku-4-5-20251001` — самая дешёвая
современная модель ($1 / $5 за million tokens), которой с большим запасом
хватает для короткого пересказа. При 50 статьях в день это выходит
примерно в **$2–6 в месяц**. Если хочется более "литературного" стиля —
поменяйте `ANTHROPIC_MODEL` на `claude-sonnet-4-6` в `.env` (дороже
примерно в 3 раза).

### Альтернативы

`RUSSIAN_PROVIDER=deepl` или `RUSSIAN_PROVIDER=libretranslate` в `.env`
переключают на дословный перевод вместо пересказа (код для этого тоже есть
в `src/russianVersion.js`) — но дословный перевод ближе к оригиналу
текстуально, а значит менее безопасен с точки зрения авторского права, чем
пересказ. `RUSSIAN_PROVIDER=mock` — тестовый режим без реальных вызовов API.

## Безопасность API

Ручное обновление отключено, пока не задан `REFRESH_TOKEN`. После задания
токен передаётся только в заголовке `Authorization: Bearer <REFRESH_TOKEN>`.
Повторные ручные запуски ограничены `REFRESH_COOLDOWN_SECONDS` (по умолчанию
60 секунд). Плановое cron-обновление и обновление при старте не используют
токен.

CORS выключен по умолчанию. Если API нужен браузерному клиенту на другом
origin, явно перечислите разрешённые адреса в `CORS_ORIGIN` через запятую.

## Комментарии и модерация

Комментарии к статьям сохраняются со статусом `pending` и появляются
публично только после одобрения. Форма защищена honeypot-полем, ограничениями
длины и in-memory rate limit по IP (`COMMENT_RATE_LIMIT_WINDOW_SECONDS` и
`COMMENT_RATE_LIMIT_MAX`).

`/admin` рекомендует вход через Google OpenID Connect. Существующая пара
`ADMIN_USER` и `ADMIN_PASSWORD` продолжает работать только как аварийная
учётная запись администратора через HTTP Basic Auth.
Дополнительные отдельные учётные записи задаются секретом
`ADMIN_ACCOUNTS_JSON` в формате массива объектов `username`, `password`,
`role` (`admin` или `editor`). Если ни одной полной учётной записи нет,
модерация закрыта. Используйте Basic Auth только по HTTPS и с уникальными
сильными паролями. Все действия модерации требуют
`Origin`, точно совпадающий с `SITE_URL`; запросы без него или с чужого сайта
получают `403`. Для Coolify за одним reverse proxy установите `TRUST_PROXY=1`,
чтобы rate limit комментариев использовал IP клиента из `X-Forwarded-For`.
Вне доверенного reverse proxy оставьте эту настройку выключенной.

### Вход редакторов через Google

Рекомендуемый production-вход использует Google OpenID Connect. Он запрашивает
только `openid`, `email` и `profile`: сайт не получает пароль Gmail и не имеет
доступа к письмам. Сервер использует `state`, `nonce`, PKCE и официальную
`google-auth-library`, проверяет подпись и audience ID-токена, `email_verified`
и allowlist. OAuth state дополнительно привязан к одноразовой HttpOnly cookie
браузера. В SQLite хранится только SHA-256 случайного токена сессии; cookie
имеет `HttpOnly`, `SameSite=Lax` и `Secure` на HTTPS.

1. В Google Cloud создайте OAuth Client типа **Web application**.
2. Добавьте точный redirect URI
   `https://finskienovosti.fi/admin/auth/google/callback`.
3. В Coolify задайте `GOOGLE_OAUTH_CLIENT_ID`,
   `GOOGLE_OAUTH_CLIENT_SECRET` и `ADMIN_GOOGLE_ACCOUNTS_JSON`.
4. Оставьте старый `ADMIN_USER`/`ADMIN_PASSWORD` только как временный аварийный
   доступ и удалите его после приёмки Google-входа.

Сессия действует `ADMIN_SESSION_HOURS` часов. Входы, отклонённые аккаунты,
выходы и редакционные действия видны в журнале аудита.

Роли разделены по принципу минимальных прав:

| Действие | `editor` | `admin` |
|---|---:|---:|
| Создавать, импортировать, править и публиковать статьи | да | да |
| Модерировать комментарии и отправлять статью в Telegram | да | да |
| Смотреть статистику, CSV, повторы и журнал | да | да |
| Безвозвратно удалять статью или комментарий | нет | да |

Запрещённая попытка редактора возвращает `403` и записывается в журнал как
`authorization.denied`. В production владельцу назначается `admin`, а всем
остальным сотрудникам — `editor`.

В админ-панели можно редактировать имя и текст любого комментария, менять его
статус, удалять комментарии, искать и редактировать статьи, создавать ручные
материалы, задавать срок закрепления и планировать публикацию на будущую дату.
Для ожидаемого финского времени задайте серверу `TZ=Europe/Helsinki`.
Автоматически пропущенный похожий материал можно проверить и опубликовать
кнопкой «Опубликовать всё равно». Статистика фильтруется по периоду до 90 дней,
категории и источнику; результат выгружается в CSV. Отдельный журнал действий
фиксирует учётную запись, действие, объект и время, но не хранит пароли или
полный текст материала.

## Анонимная аналитика и срок хранения

Для дневной статистики просмотров и реакций сервер хранит только HMAC от
IP-адреса и User-Agent вместе с датой UTC; исходные IP и User-Agent в SQLite
не записываются. HMAC включает дату, поэтому один посетитель не получает
стабильный идентификатор между разными днями. Записи старше
`ANALYTICS_RETENTION_DAYS` (по умолчанию 90 дней) удаляются при старте и
ежедневно в 00:15 UTC. Перед публичным запуском опубликуйте политику
конфиденциальности, описывающую эту статистику и комментарии.

## Ручная отправка в Telegram

В `/admin` редактор может вручную отправить статью в Telegram. Для этого нужны
обе переменные: `TELEGRAM_BOT_TOKEN` и `TELEGRAM_CHAT_ID`. Если хотя бы одна
не задана, отправка отключена и внешний запрос не выполняется. Используйте
`TELEGRAM_API_BASE_URL=https://api.telegram.org`; изменять его допускается
только для локального mock API при проверке. После успешной отправки статья
получает запись в журнале и повторная отправка кнопкой недоступна.

Настройка канала:

1. Создайте бота командой `/newbot` у официального `@BotFather`.
2. Добавьте бота администратором Telegram-канала с правом публикации сообщений.
3. Для публичного канала используйте `TELEGRAM_CHAT_ID=@channel_username`; для
   закрытого канала нужен его числовой Bot API ID вида `-100…`.
4. Сохраните токен и Chat ID только как secrets в Coolify и выполните redeploy.

Кнопка в `/admin` вызывает официальный метод Bot API `sendMessage`. Сообщение
содержит редакционную метку, русский заголовок, краткий текст и постоянную
ссылку на статью. Успех записывается по `message_id`, поэтому повторная ручная
отправка этой статьи блокируется.

## HTML-страницы

Сервер рендерит страницы напрямую из SQLite: `/` — 50 свежих новостей,
`/page/:number` — постоянные страницы архива по 50 материалов,
`/category/:slug` — категория, `/search?q=...` — поиск по всему опубликованному
архиву и `/news/:slug` — постоянная страница отдельной новости. Укажите
публичный адрес сайта в `SITE_URL`, чтобы canonical URL был корректным.

Над AI-дайджестом главной страницы показывается бегущая строка последних
одобренных комментариев. Комментарии со статусом `pending` или `rejected` не
публикуются в этой строке.

## SEO

`/sitemap.xml` содержит главную страницу, страницы архива, категории и все
опубликованные статьи, включая материалы без даты публикации.
`/robots.txt` разрешает индексацию HTML-страниц, запрещает `/api/` и содержит
абсолютный URL sitemap, построенный из `SITE_URL`.

## ⚠️ Важно: авторское право и юридическая сторона

Пересказ через ИИ (вместо дословного перевода) — более безопасный подход,
но не железная гарантия. Прежде чем публиковать такой сайт для широкой
аудитории, стоит понимать:

- Даже пересказ своими словами остаётся производным материалом от чужой
  новости. Это распространённая практика у нишевых агрегаторов, но
  **не автоматическое разрешение** на коммерческое использование в любом
  объёме.
- Обязательно держите на каждой карточке видимую ссылку на оригинал и
  название источника (в текущей вёрстке сайта это уже сделано) — читатель
  должен иметь возможность в один клик перейти к первоисточнику.
- Не пытайтесь через ИИ реконструировать полный текст платной статьи по
  крупицам — пересказ должен оставаться коротким тизером, а не заменой
  чтения оригинала.
- Если планируете расти и монетизировать проект — стоит списаться с
  редакциями (у YLE, например, есть открытые API для медиа-партнёров)
  и уточнить условия использования их RSS для агрегатора.
- robots.txt и Условия использования (Terms of Service) каждого сайта
  стоит проверить отдельно — RSS-фид, отданный публично, не всегда
  означает разрешение на любое переиспользование контента.

Это не юридическая консультация — при серьёзных амбициях по проекту
лучше свериться с юристом, знакомым с финским и европейским
законодательством об авторском праве (в т.ч. вопрос press publishers'
right в рамках копирайт-директивы ЕС, который прямо касается
агрегаторов новостей).

## Деплой

Используйте Node.js 20 или новее: это требуется текущей версией
`better-sqlite3`.

Для продакшена подойдёт любой Node.js хостинг с поддержкой
долгоживущих процессов и cron внутри процесса: Railway, Render,
Fly.io, обычный VPS с `pm2`. Serverless-платформы (Vercel Functions и
т.п.) не подойдут напрямую — там нет длительно живущего процесса для
`node-cron`, придётся выносить расписание во внешний cron-триггер.
