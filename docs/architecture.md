# Архитектурное решение: постоянный новостной архив

Дата: 2026-07-19
Статус: реализовано и развивается

## Решение

Сайт использует один сервер Node.js + Express с server-side rendering (SSR) и
SQLite как основной источник данных. Каждая опубликованная новость получает
постоянный URL. Старый JSON-кэш используется только как вход одноразовой
миграции и не является источником правды.

Главная страница, категории, архив и страницы статей рендерятся сервером. На
первом релизе JavaScript в браузере нужен лишь для небольших улучшений
интерфейса; он не отвечает за загрузку основной новостной ленты.

## Границы первого релиза

В релиз входят:

- RSS-сбор четырёх подтверждённых источников;
- AI-пересказ коротких RSS-анонсов;
- SQLite-архив статей и двухуровневая защита от повторной публикации;
- постоянные slug URL вида `/news/<slug>`;
- главная, категории и архив с пагинацией по 50 статей;
- `sitemap.xml`, `robots.txt`, canonical URL и базовые meta-теги;
- обратная ссылка на оригинальную публикацию и название источника.

За рамками ближайшего production-запуска остаются только автоматическая
публикация в Telegram, автоматическая переработка уже опубликованных
AI-пересказов и клиентская SPA-архитектура. Комментарии, редакционная админка,
анонимная аналитика, реакции и ручная публикация в Telegram уже реализованы.

Закрытая редакция использует Google OpenID Connect. Одноразовые OAuth state,
nonce и PKCE verifier хранятся в SQLite не более десяти минут; состояние
используется только один раз и привязано к HttpOnly cookie браузера. После
проверки ID-токена сервер создаёт случайную сессию, сохраняет только SHA-256
токена и отправляет браузеру HttpOnly cookie.
Google access/refresh tokens и пароль Gmail не сохраняются. Список разрешённых
email и ролей задаётся production-переменной, поэтому удаление адреса немедленно
отзывает его текущую сессию. Basic Auth остаётся только временным аварийным
механизмом до приёмки OAuth на production-домене.

## Данные

Таблица `articles` должна содержать как минимум:

| Поле | Назначение |
|---|---|
| `id` | внутренний целочисленный идентификатор |
| `source_id` | идентификатор источника (`yle`, `hs`, `il`, `is`) |
| `source_name` | отображаемое название источника |
| `original_url` | ссылка на оригинал, уникальна |
| `external_guid` | GUID RSS, если он передан источником |
| `slug` | уникальный постоянный URL-идентификатор |
| `category` | рубрика |
| `title_fi`, `summary_fi` | исходный RSS-анонс |
| `title_ru`, `summary_ru` | опубликованный пересказ |
| `translation_method` | способ получения текста |
| `prompt_version` | версия промпта для аудита |
| `published_at` | время публикации источником |
| `created_at` | время первого сохранения в архив |

Уникальность: `original_url` — первый уровень дедупликации. `slug` также должен
быть уникальным. До AI-пересказа второй уровень сравнивает заголовок и RSS-анонс
с публикациями других источников за тот же день. Только совпадение с высокой
уверенностью пропускается и записывается в `article_duplicate_log` для аудита.
Система не объединяет и не удаляет уже опубликованные постоянные страницы.

Уже опубликованная статья не перезаписывается во время обычного RSS-обновления.
Новая версия промпта применяется только к новым материалам. Ручной
пересказ/обновление в будущем должен быть отдельным админ-действием с аудитом.

## Маршруты

| Путь | Назначение |
|---|---|
| `/` | свежая лента, первые 50 статей |
| `/page/:number` | архив по страницам |
| `/category/:slug` | лента одной категории |
| `/news/:slug` | постоянная SSR-страница статьи |
| `/sitemap.xml` | карта страниц статей |
| `/robots.txt` | правила для поисковых роботов |
| `/api/health` | техническая проверка сервера |
| `/admin/login` | вход редактора через разрешённый Google-аккаунт |
| `/admin/auth/google/callback` | защищённый OAuth callback |

Старый `/api/news` можно оставить временно, но он должен читать данные из
SQLite. После перехода фронтенда он не является публичным контрактом сайта.

## Миграция

1. Добавить SQLite-слой и миграцию схемы без изменения HTTP-страниц.
2. Написать одноразовый скрипт импорта `data/latestNews.json` в SQLite.
3. Перевести RSS-сбор на запись только в SQLite и проверить дедупликацию.
4. Добавить SSR-страницы, архив и SEO-маршруты.
5. Выполнить локальный QA с mock-провайдером, затем один контролируемый запуск
   с Claude API.
6. Только после этого подключать Docker/Coolify и production-домен.

## Эксплуатационные правила

- `better-sqlite3` добавляется в `dependencies` и фиксируется lock-файлом.
- Файл SQLite хранится в persistent volume Coolify и регулярно копируется в
  резервную папку/хранилище.
- Один процесс приложения выполняет RSS-обновление; параллельные обновления
  блокируются существующим `isRefreshing`. Плановые задачи используют
  `node-cron` 4 с `noOverlap`, чтобы один и тот же cron не запускался повторно
  до завершения предыдущего выполнения.
- Журнал похожих сюжетов хранит ссылку на пропущенный оригинал, найденное
  совпадение и оценку уверенности; редактор видит его в закрытой админке и
  может явно опубликовать ошибочно определённый повтор.
- Отложенные статьи остаются черновиками до указанного времени; минутная
  серверная задача переводит их в опубликованные и записывает системное
  действие в журнал аудита.
- Административная авторизация использует Google OIDC и поддерживает несколько
  временных Basic Auth аккаунтов. В SQLite сохраняются только email/роль,
  хеши сессий и действия, но не пароли или Google-токены.
- Роль `editor` ведёт материалы, импорт, модерацию, аналитику и Telegram.
  Безвозвратное удаление статьи или комментария разрешено только `admin`;
  отказ фиксируется в журнале аудита как `authorization.denied`.
- Анонимные дневные просмотры и реакции дедуплицируются SQLite по HMAC
  посетителя; IP-адреса и User-Agent в открытом виде не сохраняются.
- Production CORS ограничивается доменом сайта или отключается, если внешний
  JSON API больше не нужен.

## Критерии готовности этапа SQLite/SSR

- повторный RSS-сбор не создаёт дубликатов;
- каждая статья открывается по постоянному URL после перезапуска сервера;
- архив и категория возвращают предсказуемую пагинацию по 50 материалов;
- `sitemap.xml` содержит канонические URL опубликованных статей;
- mock-интеграционный тест проходит без сети к AI-провайдеру;
- `.env.example` документирует все новые переменные;
- JSON-кэш не требуется для работы SSR-сайта.
